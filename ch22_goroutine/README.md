# 22장 고루틴과 동시성 프로그래밍

---

## 실행 방법

Docker 컨테이너 내부에서 다음과 같이 실행한다:

```bash
# 컨테이너 접속 (최초 1회)
make shell

# 예제 실행
go run ch22_goroutine/basic.go
go run -race ch22_goroutine/race_condition.go
go run ch22_goroutine/mutex.go
go run ch22_goroutine/waitgroup.go
go run ch22_goroutine/deadlock.go
```

> **참고**: `race_condition.go`는 `-race` 플래그를 사용하여 경쟁 상태를 탐지할 수 있다.

> **Makefile 활용**: `make run CH=ch22_goroutine` 또는 `make run CH=ch22_goroutine FILE=basic.go`

---

Go 언어의 가장 강력한 특징 중 하나는 **고루틴(Goroutine)**을 통한 동시성 프로그래밍이다. 동시성(concurrency)은 여러 작업을 논리적으로 동시에 진행하는 구조를 의미하며, 병렬성(parallelism)과는 구별된다. 병렬성은 물리적으로 동시에 실행되는 것이고, 동시성은 여러 작업을 번갈아가며 진행하여 동시에 처리되는 것처럼 보이게 하는 것이다. Go는 언어 차원에서 동시성을 지원하므로, 다른 언어에 비해 훨씬 간결하고 효율적으로 동시성 프로그래밍을 할 수 있다. 이 장에서는 고루틴의 개념부터 동시성 프로그래밍의 주의점까지 학습한다.

## 22.1 스레드란?

### OS 스레드

**스레드(Thread)**는 프로세스 내에서 실행되는 가장 작은 실행 단위이다.

- 하나의 프로세스는 여러 스레드를 가질 수 있다.
- OS 스레드는 운영체제가 직접 관리하며, 생성 비용이 크다.
- 일반적으로 스레드 하나당 **1~8MB**의 스택 메모리를 사용한다.
- 스레드 간 컨텍스트 스위칭은 커널 모드 전환을 수반하므로 비용이 높다.

### 고루틴 (Goroutine)

Go의 고루틴은 OS 스레드보다 훨씬 가벼운 **경량 스레드**이다. Go 런타임이 자체적으로 관리하므로 OS에 의존하지 않으며, 매우 적은 메모리로 시작하여 필요에 따라 동적으로 스택을 확장한다.

| 비교 항목 | OS 스레드 | 고루틴 |
|-----------|----------|--------|
| 스택 크기 | 1~8MB (고정) | 2KB (동적 확장) |
| 생성 비용 | 높음 | 매우 낮음 |
| 컨텍스트 스위칭 | 커널 모드 전환 필요 | 유저 모드에서 처리 |
| 동시 실행 가능 수 | 수천 개 | 수십만 개 이상 |
| 관리 주체 | 운영체제 | Go 런타임 |

고루틴의 스택은 2KB로 시작하지만, 필요에 따라 최대 1GB까지 동적으로 확장된다. 이 덕분에 수십만 개의 고루틴을 동시에 생성해도 메모리 부담이 적다.

## 22.2 고루틴 사용

`go` 키워드를 함수 호출 앞에 붙이면 해당 함수가 고루틴으로 실행된다. 이때 함수는 비동기적으로 실행되며, 호출한 쪽은 기다리지 않고 다음 코드를 계속 실행한다.

```go
go someFunction()     // 함수 호출을 고루틴으로 실행
go func() {           // 익명 함수도 가능
    fmt.Println("안녕!")
}()
```

### 주의점

메인 함수(`main`)가 종료되면 모든 고루틴도 함께 종료된다. 메인 고루틴은 다른 고루틴의 완료를 자동으로 기다리지 않는다.

```go
func main() {
    go fmt.Println("고루틴에서 실행")
    // 메인 함수가 즉시 종료되면 위 고루틴이 실행되지 않을 수 있음!
}
```

따라서 고루틴이 완료될 때까지 대기하는 메커니즘이 필요하다 (WaitGroup, 채널 등). `time.Sleep`으로 대기하는 방법은 실무에서는 사용하지 않는다. 고루틴의 실행 시간을 예측할 수 없기 때문이다.

### 고루틴과 클로저

고루틴에서 클로저를 사용할 때는 루프 변수 캡처에 주의해야 한다.

```go
// 잘못된 예 (Go 1.21 이전)
for i := 0; i < 5; i++ {
    go func() {
        fmt.Println(i)  // 모든 고루틴이 같은 i를 참조할 수 있다
    }()
}

// 올바른 예: 매개변수로 전달
for i := 0; i < 5; i++ {
    go func(n int) {
        fmt.Println(n)  // 각 고루틴이 고유한 값을 가진다
    }(i)
}
```

> **참고:** Go 1.22부터는 루프 변수가 각 반복마다 새로 생성되므로 이 문제가 해결되었다.

## 22.3 고루틴의 동작 방법

### M:N 스케줄링

Go는 **M개의 고루틴**을 **N개의 OS 스레드**에 매핑하는 M:N 스케줄링 모델을 사용한다. 이를 통해 적은 수의 OS 스레드로 많은 수의 고루틴을 효율적으로 실행할 수 있다.

### GMP 모델

Go 스케줄러는 GMP 모델로 동작한다.

- **G (Goroutine)**: 실행할 고루틴
- **M (Machine)**: OS 스레드
- **P (Processor)**: 논리적 프로세서 (기본값: CPU 코어 수, `runtime.GOMAXPROCS`로 설정 가능)

```
         P (Processor)
         ├── 로컬 큐: [G1, G2, G3]
         └── M (OS Thread)
              └── 현재 실행 중: G0

         P (Processor)
         ├── 로컬 큐: [G4, G5]
         └── M (OS Thread)
              └── 현재 실행 중: G6

         글로벌 큐: [G7, G8, ...]
```

**동작 원리:**
1. 각 P는 로컬 큐에서 고루틴을 꺼내 M에서 실행한다.
2. 로컬 큐가 비면 글로벌 큐에서 가져온다.
3. 글로벌 큐도 비면 다른 P의 큐에서 **훔쳐온다(work stealing)**.
4. 고루틴이 시스템 콜로 블로킹되면, M을 분리하고 새 M을 할당한다.

work stealing 알고리즘은 각 프로세서의 부하를 균등하게 분배하여 전체적인 처리량을 극대화한다. 이 알고리즘 덕분에 개발자가 직접 작업 분배를 신경 쓸 필요가 없다.

## 22.4 동시성 프로그래밍 주의점

### Race Condition (경쟁 상태)

여러 고루틴이 동시에 같은 데이터에 접근하면 **경쟁 상태**가 발생한다. 경쟁 상태는 프로그램의 결과가 고루틴의 실행 순서에 따라 달라지는 비결정적(non-deterministic) 상황을 말한다.

```go
counter := 0
for i := 0; i < 1000; i++ {
    go func() {
        counter++  // 경쟁 상태! 결과가 매번 달라진다.
    }()
}
```

`counter++`는 실제로 3단계 연산이다:
1. 메모리에서 값을 읽기 (Read)
2. 값을 증가시키기 (Increment)
3. 메모리에 값을 쓰기 (Write)

여러 고루틴이 동시에 이 단계를 수행하면 값이 덮어씌워질 수 있다. 예를 들어, 두 고루틴이 동시에 값 5를 읽고, 각각 6을 쓰면, 두 번 증가했지만 결과는 6이 된다.

### Race Detector

Go는 내장 경쟁 상태 탐지기를 제공한다. `-race` 플래그를 사용하면 런타임에 경쟁 상태를 감지하여 보고한다. 이 플래그는 개발 및 테스트 단계에서 반드시 사용하는 것이 좋다.

```bash
go run -race your_program.go
go test -race ./...
```

> **주의:** `-race` 플래그를 사용하면 프로그램 실행 속도가 느려지고 메모리 사용량이 증가하므로, 프로덕션 환경에서는 사용하지 않는 것이 일반적이다.

## 22.5 뮤텍스를 이용한 동시성 문제 해결

`sync.Mutex`는 **상호 배제(Mutual Exclusion)**를 제공한다. 한 번에 하나의 고루틴만 잠금을 획득할 수 있으며, 다른 고루틴은 잠금이 해제될 때까지 대기한다.

```go
var mu sync.Mutex
var counter int

func increment() {
    mu.Lock()         // 잠금 획득
    counter++         // 임계 영역 (한 번에 하나의 고루틴만 실행)
    mu.Unlock()       // 잠금 해제
}
```

`defer mu.Unlock()`을 사용하면 함수가 어떤 경로로 반환되든 잠금이 반드시 해제되므로, panic이 발생해도 안전하다.

### sync.RWMutex

읽기가 많은 경우 `sync.RWMutex`가 더 효율적이다. 읽기 잠금(`RLock`)은 여러 고루틴이 동시에 획득할 수 있지만, 쓰기 잠금(`Lock`)은 단독으로만 획득할 수 있다.

```go
var rwmu sync.RWMutex

func read() int {
    rwmu.RLock()          // 읽기 잠금 (여러 고루틴 동시 읽기 가능)
    defer rwmu.RUnlock()
    return counter
}

func write(val int) {
    rwmu.Lock()           // 쓰기 잠금 (단독 접근)
    defer rwmu.Unlock()
    counter = val
}
```

읽기 작업이 쓰기 작업보다 훨씬 빈번한 경우(예: 캐시, 설정 조회), `RWMutex`를 사용하면 읽기 성능이 크게 향상된다.

## 22.6 뮤텍스와 데드락

**데드락(Deadlock)**은 두 개 이상의 고루틴이 서로가 가진 잠금을 기다리며 영원히 대기하는 상태이다. Go 런타임은 모든 고루틴이 블로킹된 경우 "fatal error: all goroutines are asleep - deadlock!"을 출력하고 프로그램을 종료한다.

```
고루틴 A: Lock(mu1) → Lock(mu2) 시도 (mu2 대기)
고루틴 B: Lock(mu2) → Lock(mu1) 시도 (mu1 대기)
→ 둘 다 영원히 대기! (데드락)
```

### 데드락 방지 방법

1. **잠금 순서 일관성**: 항상 같은 순서로 잠금을 획득한다. 예를 들어, 항상 mu1 → mu2 순서로 잠그면 데드락이 발생하지 않는다.
2. **잠금 범위 최소화**: 임계 영역을 최대한 짧게 유지한다. 잠금 안에서 I/O 작업이나 네트워크 호출을 하지 않도록 한다.
3. **타임아웃 사용**: context를 사용하여 대기 시간 제한을 둔다.
4. **채널 사용**: 뮤텍스 대신 채널 기반 설계를 고려한다. Go의 철학은 "메모리를 공유하여 통신하지 말고, 통신으로 메모리를 공유하라"이다.

## 22.7 또 다른 자원 관리 기법

### sync.WaitGroup

여러 고루틴의 완료를 기다릴 때 사용한다. 내부적으로 카운터를 관리하며, `Add`로 증가, `Done`으로 감소, `Wait`로 카운터가 0이 될 때까지 블로킹한다.

```go
var wg sync.WaitGroup

for i := 0; i < 5; i++ {
    wg.Add(1)          // 카운터 증가
    go func(id int) {
        defer wg.Done() // 완료 시 카운터 감소
        fmt.Println("작업", id, "완료")
    }(i)
}

wg.Wait()              // 모든 고루틴이 완료될 때까지 대기
```

> **주의:** `wg.Add(1)`은 반드시 고루틴을 시작하기 전에 호출해야 한다. 고루틴 내부에서 `Add`를 호출하면 `Wait`가 먼저 반환될 수 있는 경쟁 상태가 발생한다.

### sync.Once

함수를 **딱 한 번만** 실행하도록 보장한다. 초기화에 유용하며, 여러 고루틴이 동시에 호출해도 안전하다. 내부적으로 뮤텍스와 원자적 연산을 사용하여 구현되어 있다.

```go
var once sync.Once
var instance *Database

func GetDB() *Database {
    once.Do(func() {
        instance = &Database{} // 이 블록은 딱 한 번만 실행됨
        instance.Connect()
    })
    return instance
}
```

### sync.Map

Go 1.9부터 제공되는 동시성에 안전한 맵이다. 일반적인 `map`에 `Mutex`를 조합하는 것보다 특정 사용 패턴(키가 한 번 쓰여지고 많이 읽히는 경우, 키가 분리된 고루틴에서 접근되는 경우)에서 더 효율적이다.

```go
var m sync.Map
m.Store("key", "value")
val, ok := m.Load("key")
m.Delete("key")
```

## 핵심 요약

| 개념 | 설명 |
|------|------|
| 고루틴 | `go` 키워드로 생성하는 경량 스레드. 스택 2KB부터 시작 |
| GMP 모델 | G(고루틴), M(OS 스레드), P(프로세서)로 구성된 스케줄링 모델 |

| `sync.RWMutex` | 읽기/쓰기 구분 잠금. 읽기는 동시 허용, 쓰기는 독점 |
| Deadlock | 서로의 잠금을 대기하며 영원히 블로킹되는 상태 |
| `sync.WaitGroup` | 여러 고루틴의 완료를 대기 (Add, Done, Wait) |
| `sync.Once` | 함수를 딱 한 번만 실행하도록 보장 |
| `sync.Map` | 동시성에 안전한 맵 (특정 사용 패턴에 최적화) |
| `-race` 플래그 | 경쟁 상태 탐지기 (`go run -race`) |

## 연습문제

### 문제 1: 병렬 합계
1부터 100까지의 합을 10개의 고루틴으로 나누어 계산하라.
- 각 고루틴은 10개의 숫자를 담당한다 (1-10, 11-20, ..., 91-100).
- `sync.Mutex`와 `sync.WaitGroup`을 사용하여 최종 합계를 구하라.

### 문제 2: 안전한 맵
`sync.Mutex`를 사용하여 동시성에 안전한 맵(SafeMap)을 구현하라.
- `Set(key, value)`, `Get(key)`, `Delete(key)` 메서드를 제공하라.

### 문제 3: Race Condition 탐지
의도적으로 경쟁 상태가 있는 코드를 작성하고, `go run -race`로 탐지하라.
그런 다음 `sync.Mutex`로 수정하여 경쟁 상태를 제거하라.

### 문제 4: 싱글톤 패턴
`sync.Once`를 사용하여 설정(Config) 객체를 싱글톤으로 구현하라.
여러 고루틴에서 동시에 `GetConfig()`를 호출해도 설정이 한 번만 로드되어야 한다.

### 문제 5: RWMutex 성능 비교
다음 두 가지 방식으로 동시 읽기/쓰기 프로그램을 작성하고 성능을 비교하라.
- 방식 1: `sync.Mutex`만 사용
- 방식 2: `sync.RWMutex` 사용
- 100개의 읽기 고루틴과 10개의 쓰기 고루틴을 실행하고, 총 실행 시간을 `time.Since`로 측정하여 비교하라.

### 문제 6: 고루틴 누수 확인
고루틴 누수(goroutine leak)가 발생하는 코드를 의도적으로 작성하고, `runtime.NumGoroutine()`으로 고루틴 수를 확인하라.
- 채널에서 읽기를 기다리는 고루틴을 생성하되, 아무도 값을 보내지 않는 상황을 만들어라.
- `runtime.NumGoroutine()`을 주기적으로 출력하여 고루틴이 증가하는지 관찰하라.
- 올바르게 수정하여 고루틴 누수를 해결하라.

### 문제 7: 동시성 카운터 (atomic 사용)
`sync/atomic` 패키지를 사용하여 뮤텍스 없이 동시성에 안전한 카운터를 구현하라.
- `atomic.AddInt64`, `atomic.LoadInt64`를 사용하라.
- 1000개의 고루틴에서 동시에 카운터를 증가시키고, 최종 값이 정확히 1000인지 확인하라.

### 문제 8: WaitGroup 카운트 오류 디버깅
아래 코드에서 문제를 찾아 수정하라.
```go
var wg sync.WaitGroup
for i := 0; i < 5; i++ {
    go func(n int) {
        wg.Add(1)
        defer wg.Done()
        fmt.Println(n)
    }(i)
}
wg.Wait()
```

### 문제 9: 타임아웃이 있는 뮤텍스
Go에는 타임아웃이 있는 뮤텍스가 내장되어 있지 않다. 채널을 활용하여 타임아웃 기능이 있는 `TryLock` 함수를 구현하라.
- `func TryLock(mu *sync.Mutex, timeout time.Duration) bool`
- 지정된 시간 내에 잠금을 획득하면 `true`, 타임아웃이 발생하면 `false`를 반환하라.

### 문제 10: 동시성 안전한 슬라이스
동시성에 안전한 `SafeSlice[T any]` 제네릭 타입을 구현하라.
- `Append(item T)`, `Get(index int) (T, bool)`, `Len() int`, `Range(fn func(T))` 메서드를 제공하라.
- `sync.RWMutex`를 사용하여 읽기/쓰기를 적절히 보호하라.

## 구현 과제

### 과제 1: 웹 크롤러 시뮬레이터
URL 목록을 받아 동시에 다운로드하는 웹 크롤러를 시뮬레이션하라.
- `time.Sleep`으로 네트워크 지연을 시뮬레이션하라 (URL마다 100ms~500ms 랜덤).
- 최대 동시 다운로드 수를 제한하라 (세마포어 패턴 사용).
- `sync.WaitGroup`으로 모든 다운로드 완료를 대기하고, 총 소요 시간을 측정하라.
- 순차 처리와 동시 처리의 실행 시간을 비교하라.

### 과제 2: 동시성 안전 캐시
만료 시간(TTL)을 지원하는 동시성 안전 캐시를 구현하라.
- `Set(key string, value interface{}, ttl time.Duration)`, `Get(key string) (interface{}, bool)`, `Delete(key string)` 메서드를 구현하라.
- 만료된 항목을 주기적으로 정리하는 백그라운드 고루틴을 실행하라.
- `sync.RWMutex`를 사용하여 읽기/쓰기를 보호하라.

### 과제 3: 작업 풀 (Worker Pool)
고정된 수의 워커 고루틴이 작업 큐에서 작업을 꺼내 처리하는 워커 풀을 구현하라.
- `type WorkerPool struct`를 정의하고, `Submit(task func())`, `Shutdown()` 메서드를 구현하라.
- 워커 수, 작업 큐 크기를 설정할 수 있게 하라.
- 모든 작업이 완료된 후 깔끔하게 종료(graceful shutdown)되도록 하라.

### 과제 4: 동시성 소수 판별기
주어진 범위의 숫자들을 여러 고루틴으로 분배하여 소수를 찾는 프로그램을 작성하라.
- `GOMAXPROCS` 값에 맞춰 고루틴 수를 설정하라.
- 각 고루틴이 찾은 소수를 안전하게 결과 슬라이스에 추가하라.
- 단일 고루틴 버전과 다중 고루틴 버전의 실행 시간을 비교하라.

### 과제 5: 동시성 파일 처리기
디렉토리에서 모든 텍스트 파일을 읽고, 각 파일의 단어 수를 동시에 계산하는 프로그램을 작성하라.
- 파일마다 고루틴을 생성하여 동시에 처리하라.
- 결과를 `sync.Map` 또는 뮤텍스로 보호되는 맵에 저장하라.
- 모든 파일의 총 단어 수와 파일별 단어 수를 출력하라.

## 프로젝트 과제

### 프로젝트 1: 동시성 다운로드 매니저
파일 다운로드 매니저를 구현하라.
- 여러 URL을 동시에 다운로드하되, 최대 동시 다운로드 수를 제한하라.
- 각 다운로드의 진행 상태(다운로드 중, 완료, 실패)를 실시간으로 출력하라.
- 다운로드 실패 시 최대 3회 재시도하라.
- 전체 진행률(완료/전체)을 표시하라.
- `sync.WaitGroup`, `sync.Mutex`, 고루틴을 조합하여 구현하라.

### 프로젝트 2: 실시간 데이터 집계 시스템
여러 데이터 소스에서 동시에 데이터를 수집하고 실시간으로 집계하는 시스템을 구현하라.
- 3개 이상의 데이터 생산자(producer) 고루틴이 랜덤 데이터를 생성하라.
- 집계자(aggregator)가 모든 생산자의 데이터를 수집하여 평균, 최솟값, 최댓값, 합계를 실시간으로 계산하라.
- 1초마다 현재 통계를 출력하라.
- 10초 후에 모든 생산자를 종료하고 최종 통계를 보고하라.
- 동시성 안전성을 보장하기 위해 뮤텍스와 WaitGroup을 적절히 사용하라.
